# Resume the exact Claude session on recreate/restart

**Date:** 2026-05-22
**Status:** Design â€” approved for implementation planning

## Summary

When the app (re)launches `claude` in a tmux pane, make it resume the *exact*
Claude Code session that belonged to that pane â€” not "the most recent session
for the directory". The app mints a UUID per work/review session, stores it on
the session row, launches `claude --session-id <uuid>`, and on
recreate/restart relaunches with `claude --resume <uuid>`. This fixes the bug
where recreating one of several Claude sessions sharing a worktree brings back a
*different* conversation.

## Motivation

The pane launch command is `cl --continue || cl`. `claude --continue` resumes
the most-recent session **for the working directory**. When multiple Claude
sessions run in the same repo/worktree, `--continue` picks whichever was most
recently active in that directory â€” not necessarily the conversation that was
running in the recreated tmux session. Result (confirmed by the user): Recreate
brings back a *different existing conversation*. `restart_session` uses the same
`--continue` and has the identical defect.

The fix: give the app authority over the session id. Claude Code documents
`claude --session-id <uuid>` (start a session with a caller-supplied UUID) and
`claude --resume <uuid>` (resume that exact id) â€” so the app can mint and own
the id, with zero footprint on the user's `~/.claude` config and identical
behavior local and remote.

## Scope

In scope:

- A per-session Claude session UUID, generated by the app for work/review
  sessions, stored on the session row.
- Launch/relaunch commands that create with `--session-id` and resume with
  `--resume`, with a safe fallback chain.
- Apply to `new_session`, `recreate_session`, `restart_session`, and
  `spawn_review`.
- Graceful fallback to today's `--continue` for legacy sessions (no stored id).

Out of scope:

- Capturing ids for Claude sessions the app did **not** launch (would need the
  SessionStart-hook approach; rejected â€” Recreate only ever rebuilds
  app-managed sessions).
- Shell sessions (`kind="shell"`) â€” they don't run Claude; unchanged.
- Any UI display of the session id (carried in the row, not shown).

## Design decisions (resolved during brainstorming)

| Decision | Choice |
| --- | --- |
| How to know the right Claude session id | **App mints it** (`claude --session-id <uuid>`), rather than discovering it via a SessionStart hook. |
| Storage | New nullable column `claude_session_id` on `sessions` (migration 009). |
| Relaunch command | `cl --resume '<id>' 2>/dev/null \|\| cl --session-id '<id>' \|\| cl` (idempotent create-or-resume). |
| Legacy sessions (no id) | Fall back to today's `cl --continue \|\| cl`. |
| Which lifecycle ops | new_session, recreate_session, restart_session, spawn_review. |

## Background (verified Claude Code mechanics)

- `claude --session-id <uuid>` starts a session with a caller-supplied UUID
  (must be a valid UUID). Documented:
  https://code.claude.com/docs/en/cli-reference.md
- `claude --resume <uuid>` resumes that exact session id (works for
  caller-supplied ids). Documented: https://code.claude.com/docs/en/sessions.md
- Behavior of re-running `--session-id <same-uuid>` is undocumented, so we do
  **not** rely on it â€” we use `--resume` first and fall back to `--session-id`
  only when no transcript exists.
- Sessions survive abrupt termination (tmux `kill-session` / SIGKILL) and remain
  resumable; transcripts live at `~/.claude/projects/<encoded-cwd>/<id>.jsonl`.

## Architecture

### Data model

- **Migration `009_session_claude_id.sql`:**
  `ALTER TABLE sessions ADD COLUMN claude_session_id TEXT;` (nullable, default
  NULL).
- **`store.rs`:** `SessionRow` gains `claude_session_id: Option<String>`. All
  `SELECT`/row-mapping sites for sessions read the new column. Add
  `set_claude_session_id(&self, id: i64, uuid: &str) -> Result<(), rusqlite::Error>`.
  **`upsert_session` must not write `claude_session_id`** â€” reconcile doesn't
  know it, and its `ON CONFLICT DO UPDATE` must leave the column untouched so a
  stored id survives reconciliation.

### Launch-command builder (`tmux.rs`)

Replace the static `pane_command()` with:

```rust
/// The pane command for a Claude ("work"/"review") session.
/// With a known session id: resume it, else create it under that id, else a
/// bare `cl` â€” an idempotent create-or-resume. Without an id (legacy rows):
/// today's most-recent-for-cwd behavior.
pub fn pane_command_for(claude_session_id: Option<&str>) -> String {
    let tail = "exec ${SHELL:-/bin/zsh} -l";
    match claude_session_id {
        Some(id) => format!(
            "cl --resume '{id}' 2>/dev/null || cl --session-id '{id}' || cl; {tail}"
        ),
        None => format!("cl --continue || cl; {tail}"),
    }
}
```

The id is a generated UUIDv4 (`[0-9a-f-]` only), single-quoted here and
validated via `validate::claude_session_id` before it ever reaches this
builder (defense-in-depth; a tampered DB value can't inject shell).
`shell_pane_command` is unchanged.

Note: any current callers of `pane_command()` (e.g. `spawn_review`) switch to
`pane_command_for(...)`.

### Service layer (`service/sessions.rs`)

- **`new_session_inner`** (non-shell): `let claude_id = uuid::Uuid::new_v4().to_string();`
  build `pane_command_for(Some(&claude_id))`, launch, `reconcile_one_host`,
  then `store.set_claude_session_id(row.id, &claude_id)`. Shell sessions keep
  `shell_pane_command(start_command)`. A failed `set_â€¦` is **soft** (log via
  `eprintln!`; the session is live and merely loses exact-resume on a future
  recreate).
- **`recreate_session`**: extend the existing helper to
  `recreate_pane_command(kind: &str, claude_session_id: Option<&str>) -> String`:
  `shell` â†’ `shell_pane_command(None)`; else â†’ `pane_command_for(claude_session_id)`.
  Pass `sess.claude_session_id.as_deref()`. The id is unchanged across a
  recreate, so no row update is needed.
- **`restart_session`**: read the row's `kind` **and** `claude_session_id`,
  build the pane command via the same id-aware path, then `respawn_pane`. Fixes
  restart's identical wrong-session bug.
- **`spawn_review`**: generate + store a UUID for the review session (it runs
  Claude), launching via `pane_command_for(Some(&uuid))` and calling
  `set_claude_session_id` on the tagged review row.
- The MCP spawn path routes through `new_session`, so it inherits this with no
  extra change.

### Validation (`validate.rs`)

Add `claude_session_id(value: &str) -> Result<(), IpcError>`: accept a
canonical UUID (36 chars, lowercase hex + hyphens in `8-4-4-4-12` shape),
reject anything else. Used wherever a stored id is interpolated into a launch
command.

### Frontend

- `src/lib/sessions.ts`: add `claude_session_id: string | null` to the
  `SessionRow` interface (backend serializes snake_case). Carried through the
  store merge; not displayed.

## Error handling & edge cases

- **Legacy sessions** (created before this feature, `claude_session_id` NULL):
  recreate/restart fall back to `cl --continue` â€” today's behavior, no
  regression.
- **Runtime resume failure** (transcript cleaned up after 30 days, or corrupt):
  the `|| cl --session-id '<id>' || cl` tail recreates under the same id (or a
  bare session), so the pane always comes up.
- **DB write failure** when storing the id: soft-fail (logged); the session is
  unaffected, only future exact-resume is lost.
- **Injection:** ids are app-generated UUIDs, validated and single-quoted; no
  user-supplied value reaches the launch string.

## Testing

- **Store round-trip:** `set_claude_session_id` then `get_session_by_id`
  returns the stored id; a fresh session has `None`.
- **`upsert_session` preserves the id:** set an id, run an upsert for the same
  `(host, tmux_name)`, confirm `claude_session_id` is unchanged.
- **`pane_command_for`:** `Some(id)` â†’ string contains `--resume '<id>'`,
  `--session-id '<id>'`, and the bare `cl` fallback; `None` â†’ `cl --continue`
  form.
- **`recreate_pane_command(kind, id)`:** `shell` â†’ shell command; `work` +
  `Some(id)` â†’ resume form; `work` + `None` â†’ continue form.
- **`validate::claude_session_id`:** accepts a valid v4 UUID; rejects empty,
  wrong length, non-hex, and shell-metacharacter strings.
- **Manual smoke:** create two work sessions in the same worktree; recreate one
  and confirm it returns to *its own* conversation, not the other's; recreate a
  legacy session (no stored id) and confirm it still comes up via `--continue`.

## Known limitations

- Only sessions the app launches get a managed id. Sessions started manually
  inside a pane aren't tracked â€” but Recreate only ever rebuilds app-managed
  sessions, so this doesn't affect the fix.
- The first launch performs a `cl --resume` that fails fast before
  `cl --session-id` creates the session â€” a negligible one-time cost that keeps
  a single, idempotent command form across new/recreate/restart.
