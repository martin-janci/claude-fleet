# Release automation & docs refresh — design

Date: 2026-05-23
Status: approved (design), pending implementation plan

## Problem

`claude-fleet` has no release automation. The version (`0.2.0`) is duplicated in
four places that drift independently, there is no `CHANGELOG.md`, and there is
no release/tag flow. Commits already follow Conventional Commits (~29 of the
last 40), so the inputs for automation exist but are unused. Documentation is
partly stale (CLAUDE.md reports ~5,100 LOC Rust / ~6,600 frontend; actual is
~15,700 / ~12,900) and there is no contributor-facing release guide.

## Goals

1. A CI release-PR bot that, from Conventional Commits, proposes version bumps
   and a changelog, and on merge tags + publishes a GitHub Release.
2. Keep the version synchronized across all files that carry it.
3. Auto-generate a control-API / command reference from source, gated in CI so
   it cannot go stale.
4. Publish rustdoc to GitHub Pages on each release.
5. Refresh CLAUDE.md, README.md, and add a RELEASING guide.

## Non-goals

- Building, signing, or distributing app binaries (`.dmg`/`.app`). Releases are
  **tag + GitHub Release with notes only**.
- Auto-tag-on-merge without a human gate (rejected: the release-PR bot keeps a
  review checkpoint).
- Changing the existing CI test/check jobs.

## Decisions (from brainstorming)

- **Trigger model:** CI release-PR bot (release-please).
- **Release scope:** tag + GitHub Release only, no binaries.
- **Docs scope:** all four — refresh CLAUDE.md, refresh README.md, add a
  release/contributing doc, auto-generate docs from code.
- **Auto-doc target:** both a generated MCP/command reference (with CI staleness
  check) and rustdoc published to GitHub Pages.
- **Staleness gate:** CI **fails** when the committed reference is out of date
  (not warn-only). Editing a tool description requires regenerate-and-commit.

## Component A — Release-PR bot (release-please)

- `.github/workflows/release-please.yml`, triggered on `push` to `main`.
- `release-please-config.json` + `.release-please-manifest.json`, bootstrapped at
  the current `0.2.0`.
- Behavior: maintains an open **"chore: release X.Y.Z"** PR computing the next
  semver from commit types, updating all version files, and rewriting
  `CHANGELOG.md`. Merging the PR creates the git tag and a GitHub Release with
  the generated notes.
- Changelog section mapping: `feat`→Features, `fix`→Bug Fixes, with
  `docs`/`refactor`/`perf` shown and `chore`/`test`/`ci` hidden.
- Permissions: workflow needs `contents: write` and `pull-requests: write`.

## Component B — Version sync across all four files

The version appears in `package.json`, `src-tauri/tauri.conf.json`,
`src-tauri/Cargo.toml`, **and** `src-tauri/Cargo.lock`.

Plan: use release-please **`rust`** release-type rooted at `src-tauri/`, which
updates `Cargo.toml` and `Cargo.lock` natively, plus `extra-files` entries with
the JSON updater for `package.json` (`$.version`) and
`src-tauri/tauri.conf.json` (`$.version`).

**Primary technical risk.** The exact updater wiring (release-type root path,
`extra-files` path resolution relative to repo root, and confirming `Cargo.lock`
is updated by the `rust` type for this layout) must be verified during
implementation. The plan MUST include a check that a simulated bump moves the
version in **all four** files together; if the `rust`-type approach cannot drive
`Cargo.lock` cleanly here, fall back to a generic updater on `Cargo.lock`'s
package entry.

## Component C — Generated MCP / command reference + CI staleness gate

- A generator, runnable via `cargo run` (a dedicated bin target under
  `src-tauri/`), constructs `FleetTools::tool_router()` in-process and serializes
  each tool's **name, description, and input schema** to a generated markdown
  file (e.g. `docs/control-api-reference.md`). It also emits the Tauri command
  list parsed from the `tauri::generate_handler!` invocation in
  `src-tauri/src/lib.rs`. Building the router in-process means the reference
  cannot drift from the actual tool definitions.
- The generated file carries a "do not edit by hand — generated by …" header.
- CI adds a step that regenerates the file and runs `git diff --exit-code`; a
  non-empty diff fails the build, forcing the committed reference to stay
  current.
- `docs/control-api.md` (hand-written narrative) links to the generated
  reference; the two are kept distinct (prose vs. generated table).

## Component D — rustdoc → GitHub Pages

- `.github/workflows/docs.yml` (or a Pages job) runs on Release `published`:
  `cargo doc --no-deps`, then uploads and deploys via
  `actions/upload-pages-artifact` + `actions/deploy-pages`.
- Permissions: `pages: write`, `id-token: write`.
- **Manual prerequisite:** GitHub Pages must be enabled for the repo (Settings →
  Pages → source = GitHub Actions). Documented in RELEASING.md.

## Component E — Docs refresh

- **CLAUDE.md:** correct LOC (~15,700 Rust / ~12,900 frontend), note landed
  features (background sessions, reconcile tick, `fleet_health`, session
  timeline/`session_history`), and add a "Releasing" pointer to RELEASING.md.
- **README.md:** refresh overview + build/run, add CI and version badges, and
  link the rustdoc Pages site and the generated control-API reference.
- **docs/RELEASING.md:** Conventional-Commit rules, how release-please proposes
  and merges releases, how to cut a release, how the four version files stay in
  sync, and the one-time GitHub Pages enablement step.

## Data flow & ordering

1. Developer merges Conventional-Commit PRs to `main` (CI staleness gate from C
   already enforced on every PR).
2. release-please (A/B) maintains the release PR with bumped versions + changelog.
3. Maintainer merges the release PR → tag + GitHub Release.
4. Release `published` triggers rustdoc deploy (D).

Implementation order: E-docs and C can land first (independent, low-risk); A/B
together (share config); D last (depends on Pages enablement).

## Testing

- **C** is self-testing: the staleness gate is the test, run on every PR.
- **A/B:** validate config with a release-please dry run / `--dry-run` or by
  inspecting the first bot PR; assert all four version files change in lockstep.
- **D:** confirmed by a successful Pages deploy on the first release after enablement.
- Existing `ci.yml` jobs (rust + frontend) remain unchanged and must stay green.
